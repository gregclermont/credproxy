# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository status

The product (codename "credproxy") is a transparent egress proxy for workspace containers — LLM-agent sandboxes, CI runners, dev shells, batch jobs. The scheme-aware credential-injection model — the typed-scheme model that replaced the original substring-replace — is **fully implemented**: substitute family (`bearer`/`basic`/`body`), sign family (`sigv4`), and re-seal family (`oauth2-reseal` — mints the token-endpoint response into a TTL'd dynamic placeholder so even the short-lived token stays out of the workspace). The sandboxed Starlark escape hatch (`proxy/starlark_runtime.py`, via the `starlark-pyo3` proxy dependency) is wired through the scripted-injector authoring contract — an injector with `scheme = "script"` declares `family`/`slots`/`location` (+ an `api` version) in TOML and names a `.star` file (builtin examples in `cli/credproxy_cli/builtin/scripts/`); the CLI pushes the script source, the proxy compiles it into a `ScriptedScheme`. The script primitive API is the flat/implicit-ctx "option B" model: zero-arg `on_request()`/`on_response()` hooks, flat `req_*`/`resp_*` primitives reading a contextvar-bound ctx (no ctx threading), carrier crypto (so SigV4-class multi-round signing is expressible), `jwt_encode_sign`/`json_*`, and `mint`/`mint_into_json` for re-seal; builtin `ovh`/`jwt-bearer`/`oauth-reseal` worked examples ship and are tested. All are useful background; divergence from earlier docs is expected (learn-by-building). CLAUDE.md (this file) and the code are the living source of truth. **Pre-release — there are no users, so there is no backward-compatibility or migration burden: optimize purely for the best end-state design.** Rename, remove, or restructure freely (config formats, CLI flags, file layouts, on-disk state) without compat shims, deprecation aliases, or churn worry — just update the call sites, tests, and docs to match. Never keep a worse design to avoid churn. The repo holds the proxy core under `proxy/`, the host CLI at `bin/credproxy` and `bin/credp`, the CLI package under `cli/credproxy_cli/`, tests under `tests/`, and `docs/`.

## Big-picture architecture

The product is **two containers that must stay separated**:

1. **Proxy container** (Linux, requires `NET_ADMIN`): owns the netns, installs iptables rules, runs two listeners — mitmproxy on `127.0.0.1:39999` (transparent intercept) and a single aiohttp HTTP API on `0.0.0.0:39998` that serves both workspace-facing bootstrap routes and host-facing admin routes. iptables redirects sentinel-IP `:80` to the HTTP listener and everything-else-TCP to mitmproxy. The HTTP listener is port-published to the host as `127.0.0.1:<ephemeral port>` (Docker assigns a free port per proxy container so multiple workspaces run simultaneously without conflict); workspace reaches it through the sentinel redirect or directly via `127.0.0.1:39998` in the shared netns.
2. **Host CLI** (`bin/credproxy`, Python; Go later): a workspace manager. A credproxy *workspace* is a named, persistent pair of containers — a proxy container and a workspace container sharing one netns — created once, then started/stopped explicitly; multiple workspaces run concurrently. Config is TOML, one file per workspace at `$XDG_CONFIG_HOME/credproxy/workspaces/<name>.toml`. State lives at `$XDG_STATE_HOME/credproxy/workspaces/<name>/`. The host port is ephemeral and resolved at call time via `docker port`; it is never persisted. Mount-target paths are sourced from the proxy image's `ENV` declarations (read via `docker inspect`'s `Config.Env`), so the image is the single source of truth for its own API; the CLI hardcodes only its own conventions (image tag, workspace storage layout, container/volume naming, default workspace image).

The workspace container is **the user's** image — never modified, never granted privilege. This "bring your own image" constraint is load-bearing. See `docs/workspace.md` for the constraints joining the proxy's netns imposes.

**SELinux (Fedora/RHEL hosts):** host bind mounts otherwise get denied under enforcing SELinux. Split by trust: the proxy stays confined and relabels only its own mounts (token `:Z`, dev-source `:z`); the workspace runs with `--security-opt label=disable` (distrobox/toolbx style) so the user's bind-mounted project dirs work **without** relabeling/mutating them. No-op on non-SELinux hosts. Validated on Fedora rootless podman. See `docs/workspace.md`.

Traffic flow: workspace egress → iptables OUTPUT in shared netns → REDIRECT to mitmproxy (or to the HTTP API for sentinel:80) → SNI peek → either substitute-placeholder-and-forward (terminate TLS) or passthrough (`client_hello.ignore_connection = True`).

**Configuration flow**: each workspace has its own bearer token at `$XDG_STATE_HOME/credproxy/workspaces/<name>/auth.token` (mode 0644), generated by `credproxy workspace create` or `start` if absent and bind-mounted read-only into that workspace's proxy at `/run/secrets-ro/auth.token`. The proxy reads it fresh on every `/admin/config` request — no staging copy, no in-memory snapshot — so rotating the host file takes effect on the next request without a proxy restart. Config lives on tmpfs at `/run/secrets/config.json`, written by `POST /admin/config`. Lifecycle: the token is host-owned and survives everything; the tmpfs config does **not** survive a `docker stop`/`start`, so `credproxy workspace NAME start` always re-materializes and re-pushes bindings once the proxy is healthy. `credproxy dev reload` (SIGHUP) re-execs python in place and the tmpfs survives that, so pushed config persists across a reload.

**Bindings**: a *binding* ties an injector (how a credential is shaped into a request — which typed *scheme* the proxy runs) to a provider (where the value comes from), scoped to one or more hosts. Each host is a literal (exact match, the fast dict path) or a glob pattern containing `*` (matched via `proxy/hostmatch.py`; `*` spans dots, so `*.amazonaws.com` covers every AWS regional endpoint with one binding — the sigv4 scheme reads region/service per-request). Patterns are validated strictly (two rightmost labels must be literal: `*.example.com` ok, `*.com`/`*` rejected); the CLI mirrors that validation in `core/hostmatch.py` so a bad pattern fails at `binding add`. The host CLI resolves each binding's secret(s) from its provider at `start`/`apply`/`binding test` time and pushes the resolved wire config (`{"bindings": [{name, hosts, scheme, params, secret:{slot:value}, placeholder, env}]}`) to `/admin/config`; the proxy dispatches on `scheme` (substitute `bearer`/`basic`/`body`, sign `sigv4`, re-seal `oauth2-reseal`, or `script` for a sandboxed Starlark injector). Re-seal additionally mints the token-endpoint response into a TTL'd runtime substitution layer (dynamic placeholders), so the swap set is `static pushed config + runtime-augmentable layer`, never baked immutable at push time. The provider exec protocol is batch-native (`{op:"get", secrets:[refs]}` → `{values:{ref:val}}`), and the CLI batches across bindings: at resolve time (`start`/`apply`/`binding test`) it groups every binding's refs by provider and invokes each provider **once** with the deduped union, so a provider with a costly setup (a vault that must unlock, a session that must authenticate) pays that cost once per resolve rather than once per binding (`resolve_secrets` in `core/bindings.py`). The real secret never enters the workspace. The inward `/setup` endpoint exposes only least-disclosure fields (name, placeholder, env, scheme, params, hosts — never secret values/provider/secret-id) so the workspace can self-configure. The scheme model lives in the code (`proxy/schemes.py`, `proxy/starlark_runtime.py`) and `docs/injectors.md`.

**Placeholder materialization**: missing `name` and `placeholder` fields are generated (auto-name rule: `<injector>-<provider>[-N]`; placeholder from the injector's pattern) and written back into the TOML file via surgical text edits that preserve comments. The TOML file is the single source of truth; no in-memory state diverges from disk.

## Threat model

- **Workspace container**: cannot read the host filesystem, so cannot read `$XDG_STATE_HOME/credproxy/workspaces/<name>/auth.token`. Can hit `/admin/*` endpoints over the shared netns and gets 401 without the token. No window in which `/admin/config` is unauthenticated.
- **Browser on host**: blocked by Chrome's Private Network Access (we never set `Access-Control-Allow-Private-Network`) plus the `fetch_metadata_guard` middleware (rejects requests with `Sec-Fetch-Site: cross-site`/`same-site`). Both layers act before any handler runs.
- **Other host users on a multi-user host**: can read `auth.token` (mode 0644 so the in-container uid 31337 can read it through the bind mount) and forge admin requests. Damage ceiling is DoS-or-config-replace — the user's secrets live in op://, keychain, etc., and only enter the proxy through bearer-authenticated config pushes. Documented limitation; credproxy is a single-user dev workstation tool.
- **Same-user malicious process**: out of scope (already has access to ssh keys, env vars, etc.).

## Architecture decisions that should not be casually reversed

These still apply — worth surfacing because they will tempt reconsideration:

- **Two-container shape is forced**, not chosen — netfilter must run in the same kernel as the traffic, and on macOS/Windows that kernel is inside Docker Desktop's VM. A host process cannot install iptables there. Don't propose collapsing to a single host process.
- **Transparent capture of all TCP**, not port-based selection. The product promise is "every tool works"; selective capture leaks edge cases.
- **SNI-based intercept decision**, not IP-based. CDN IP reuse breaks IP rules.
- **HTTP/3 dropped at netfilter** to force TCP fallback, not intercepted. mitmproxy QUIC is experimental.
- **IPv6 dropped entirely.**
- **Bootstrap over plain HTTP from inside the netns is fine** — no eavesdropper exists on shared loopback/link-local. This resolves the chicken-and-egg of trusting the trust source. Don't add TLS or auth to the bootstrap routes.
- **Single HTTP listener for admin + bootstrap.** Bearer auth gates `/admin/*`; bootstrap routes are open. Browsers are kept out by PNA + Sec-Fetch-Site, not by a separate listener or a separate iptables rule. Don't re-split.
- **Host-owned bearer, bind-mounted into the proxy.** The host-side `auth.token` (per-workspace, under `$XDG_STATE_HOME/credproxy/workspaces/<name>/`) is the source of truth; the proxy reads it directly from the bind mount on every admin request — no staging copy, no in-memory snapshot, rotation works without restart. Don't reintroduce TOFU or in-container token generation.
- **Providers are host-side executables, protocol is parent-agnostic.** A provider is a host executable speaking a tiny stdin/stdout JSON protocol (versioned, `op`-keyed). Today the CLI execs it; a future daemon could. Don't embed secret fetching inside the proxy or couple the protocol to the CLI. See `docs/providers.md`.
- **Push model: the CLI resolves secrets and pushes resolved values to the proxy.** The proxy never calls a provider; the host CLI fetches each secret at `start`/`apply`/`binding test` and POSTs resolved literal values to `/admin/config`. Don't move provider invocation into the proxy.
- **Placeholder materialization written back to the config file.** Generated names and placeholders are inserted as surgical TOML edits (preserving comments and ordering); the file is the single source of truth. Don't hold generated values in memory or a separate state file.
- **Ephemeral proxy host port, resolved at call time.** The proxy container is launched with `-p 127.0.0.1:0:39998` (Docker assigns a free port); the CLI resolves it via `docker port` at each call rather than persisting it. This is what allows multiple workspaces to run concurrently without port conflicts. Don't hardcode the host port or cache it.
- **Strict-by-default two-surface CLI.** `credproxy` is strict (explicit names, no defaults, no prompts, scriptable); `credp` is the human alias (`credproxy --loose`). Convenience lives in the alias, not the default binary behavior, so scripts that omit the flag fail safe. Don't flip the default.
- **Proxy container holds the proxy core; host plugins (future) handle host-touchy things.** Don't push host-touchy logic into the proxy to "simplify"; it breaks cross-platform.

## Non-goals (don't accidentally implement)

- HTTP/3/QUIC interception, IPv6, DNS interception, hostname-based egress allowlisting, process attribution (PID), cert-pinning workarounds, mTLS injection, multi-workspace-per-proxy, bypass-resistance against an adversarial workspace. credproxy is a developer convenience boundary, not a hardened jail.
- Multi-user host support: documented limitation, not a feature.
- Auto-env-var push convenience (the proxy pushing conventional env vars into the workspace environment directly): deferred. The pull model via `/setup` works; the push layer is a future layering on top.

## Key constants

All declared as `ENV` in `proxy/Dockerfile` — single source of truth for both the in-container code (via the inherited process environment) and the host CLI (via `docker inspect`'s `Config.Env`). Bumping any value is an image/CLI API break: rebuild after with `credproxy dev build`.

In-container runtime values (read by `entrypoint.sh` and `constants.py`):

- `CREDPROXY_MITMPROXY_UID=31337` — mitmproxy runs as this uid; `useradd` in the Dockerfile and the iptables `-m owner --uid-owner` rule depend on it (prevents redirect loop on mitmproxy's own outbound).
- `CREDPROXY_PROXY_PORT=39999` — mitmproxy transparent-intercept bind port. Picked unusual to minimize collision with workspace-side dev tools.
- `CREDPROXY_SENTINEL_IP=169.254.1.1` — link-local for the workspace-facing endpoint, resolved as `proxy.local` from the workspace side. iptables redirects `<sentinel>:80` to `CREDPROXY_HTTP_PORT`.

CLI/image contract (also read by `bin/credproxy` from `Config.Env`):

- `CREDPROXY_HTTP_PORT=39998` — merged HTTP API bind port (admin + bootstrap). CLI publishes it as an ephemeral host port (`-p 127.0.0.1:0:39998`) and resolves the actual port via `docker port` at call time.
- `CREDPROXY_TMPFS=/run/secrets` — CLI sets up `--tmpfs` here with `mode=1777` (so the proxy's unprivileged uid can write `config.json`, including after a `docker start`); `admin.py` writes `config.json` to this directory.
- `CREDPROXY_TOKEN_PATH=/run/secrets-ro/auth.token` — CLI bind-mounts the host token here; `admin.py` reads from this exact path.
- `CREDPROXY_SOURCE=/opt/proxy` — CLI bind-mounts `proxy/` here for dev (live edits + `credproxy dev reload`); also the Dockerfile `WORKDIR`.

## Storage layout

XDG-based. Override with the XDG env vars (works for tests too).

```
$XDG_CONFIG_HOME/credproxy/                      # default ~/.config/credproxy/
  workspaces/<name>.toml                          # per-workspace config
  injectors/<name>.toml                           # user injector registry (tier 1)
  providers/<name>  (or <name>/run)               # user provider registry
  scripts/<name>.star                             # user script registry
  presets/<name>.toml                             # user preset registry

<repo>/profile/  (or $CREDPROXY_PROFILE_DIR)      # org overlay (tier 2), upstream-empty
  workspace.template.toml                         # scaffold override
  injectors/ providers/ scripts/ presets/         # registry overrides/additions

cli/credproxy_cli/builtin/                        # upstream defaults (tier 3)
  workspace.template.toml                         # the scaffold `create` stamps
  injectors/ providers/ scripts/ presets/         # builtin definitions

$XDG_STATE_HOME/credproxy/                        # default ~/.local/state/credproxy/
  default-workspace                               # current-default pointer (loose surface)
  workspaces/<name>/
    auth.token                                    # bearer token (mode 0644)
    setup_done                                    # container id that last completed setup
    applied-spec.json                             # last applied container spec (for drift)
    applied-bindings.json                         # last pushed bindings metadata (no secrets)
    sessions/<pid>                                # pidfiles for auto-stop tracking
```

**Three-tier resolution (the fork/customization seam).** Every customizable asset resolves through one ordered search path, most specific first: **user** (`$XDG_CONFIG_HOME/credproxy/`, per-machine) → **profile** (the org overlay: `$CREDPROXY_PROFILE_DIR` or `<repo>/profile/`) → **builtin** (upstream defaults in `cli/credproxy_cli/builtin/`). A same-named file in a higher tier shadows the lower; a new name adds. This is `paths.layered_dirs()` for the registries (injectors, providers, scripts, presets) and `paths.resolve_singleton()` for the one singleton, `workspace.template.toml` — a **literal** workspace config the scaffold stamps (only `{name}` substituted). The default *workspace* image is therefore just the `image` line in that template (no separate knob); `image` is mandatory in `load_config` (no built-in default — the scaffold always writes one). The *proxy* image tag and the `home` fallback are plain engine constants in `paths.py` (`IMAGE_TAG`, `DEFAULT_HOME`), deliberately **not** customizable for now. The profile overlay lets an org customize the scaffold + definitions **without editing engine code or maintaining a conflicting fork** — upstream ships `profile/` empty, so a fork only ever adds files there (its whole diff lives in `profile/`) and `CREDPROXY_PROFILE_DIR` gives a no-fork path. See `docs/forking.md`.

## Commands

Two entry points:

- `bin/credproxy` — **strict surface**: explicit workspace names everywhere, no default resolution, no prompts, no aliases, scriptable. The contract.
- `bin/credp` — **loose surface** (`credproxy --loose`): adds default-workspace resolution (announced on stderr), short aliases, and a confirmation gate on destructive-and-implicit actions. The human entry point.

`--json` is orthogonal: available on both surfaces; errors serialize as JSON, streaming commands (`logs`) emit JSON-lines.

**Workspace commands (canonical):**

- `credproxy workspace create NAME` — scaffold `<name>.toml` + `auth.token`. Does not start anything. The scaffold sets a concrete `image` (the builtin/overlay `workspace.template.toml`); to use a different image, edit the generated file. On the loose surface only, if no default workspace is set yet, the new one becomes the default (announced); it never overrides an existing default.
- `credproxy workspace use NAME` — set the default workspace pointer (loose surface only).
- `credproxy current` — print the default workspace pointer (`{"default": ...}` under `--json`); the read-only companion to `use`.
- `credproxy workspace list [FILTER]` (or `credproxy list`) — workspaces with running status and image; current default marked.
- `credproxy workspace NAME enter [-- CMD...]` — hot path: start if needed, `docker exec` into the workspace. With no `-- CMD` it runs the config `shell` (default a login shell, `bash -l` — entering is "logging in", the ssh model); `-- CMD` runs a bare, non-login command (the ssh `host cmd` model). Tracks auto-stop sessions. The primary verb.
- `credproxy workspace NAME start` — (re)start the proxy container, wait for `/health`, push bindings config, then (re)start the workspace container. Auto-creates if missing. Recreates a container when its spec drifts (proxy: image changed; workspace: image/home/mounts/env/setup or proxy netns peer changed, tracked via `credproxy.spec` label).
- `credproxy workspace NAME stop` — `docker stop -t 1` both containers; kept, not removed.
- `credproxy workspace NAME recreate [--proxy] [--reset-volume NAME ...]` — force-rebuild the workspace container from a clean slate (re-runs `setup`), then start it. Unlike `delete`, preserves all managed volumes, config, auth token, and state — only the container is replaced. Default rebuilds the **workspace container only** (keeps the running proxy + its CA, faster); `--proxy` (alias `--all`) also recreates the proxy, regenerating its CA (full re-bootstrap; the workspace re-bootstraps CA trust). `--reset-volume NAME` (repeatable) *also* wipes that managed volume (e.g. `--reset-volume home`, re-seeded from the image) while keeping config/token/state — bind/profile host-path mounts are untouched. It's the one recreate mode that destroys data, so it's gated like `delete` (confirm on an implicit default in loose mode; `--yes` bypasses). Implemented as `docker rm -f` the target(s) [+ `docker volume rm` the named volume(s)] + `start` (so it reuses the health-wait, config push, and setup re-run).
- `credproxy workspace NAME delete [--keep-volumes]` — remove both containers, the workspace's managed volumes (by name prefix), config file, and state dir; `--keep-volumes` preserves the volumes. In loose mode, prompts when targeting the default workspace.
- `credproxy workspace NAME apply` — best-effort reconcile: bindings drift → live re-push to the running proxy; container-spec drift (image/home/mounts/env/setup) → deferred with a restart hint. Reports what was applied vs. deferred.
- `credproxy workspace NAME inspect` — config + running state + host port + binding summary + itemized drift against applied-spec/applied-bindings.
- `credproxy workspace NAME config [--declared]` — dump the container-side config. Default `effective` (every field with its in-effect value, all defaults filled — including the enter-time `workdir`→home and `enter_prelude`→shim defaults that `inspect` leaves null); `--declared` shows only what's literally in the TOML. `--json` on both.
- `credproxy workspace NAME edit` — open `<name>.toml` in `$VISUAL`/`$EDITOR` (default `vi`), then validate it; warns if the edit left it invalid, else hints `apply`/`start`. Sugar over editing the file directly; interactive, so no `--json`.
- `credproxy workspace NAME logs` — `docker logs -f` the proxy container.

**Binding commands (through a workspace):**

- `credproxy workspace NAME binding add --injector INJ --provider PROV --secret REF --host HOST [--host HOST...] [--name NAME] [--placeholder PH] [--env VAR]` — `--secret` repeats as `SLOT=REF` for a multi-slot secret (bare `REF` = single-slot).
- `credproxy workspace NAME binding add --preset PRESET --provider PROV --secret REF` — generate a coordinated binding set (e.g. `github`: bearer@api + basic@github/ghcr sharing one placeholder). The preset owns name/placeholder/env/host. A preset may carry a default provider/secret: `github` defaults `--provider gh-cli` and `--secret github.com`, so a bare `binding add --preset github` wires all three off an existing `gh` login. The default secret applies only for the default provider (a ref's meaning is provider-specific), so a non-default `--provider` still requires `--secret`.
- `credproxy workspace NAME binding remove BINDING_NAME`
- `credproxy workspace NAME binding list`
- `credproxy workspace NAME binding test [BINDING_NAME]` — dry-run fetch via provider (reports secret length, not value); exit 1 if any fail.
- `credproxy workspace binding test --provider PROV --secret REF [--injector INJ]` — ad-hoc mode: test a definition *before* it is bound. No workspace is resolved or required (the workspace name is ignored); validates that the provider/injector resolve and the fetch succeeds.

**Definition commands:**

- `credproxy injector scaffold NAME` — copy the builtin `bearer` template to `$XDG_CONFIG_HOME/credproxy/injectors/<name>.toml`.
- `credproxy injector list`
- `credproxy provider scaffold NAME` — copy the builtin `env` template to `$XDG_CONFIG_HOME/credproxy/providers/<name>`.
- `credproxy provider list`

**Loose aliases** (loose surface / `credp` only):

`credp enter [NAME]`, `credp start [NAME]`, `credp stop [NAME]`, `credp recreate [NAME] [--proxy] [--reset-volume NAME]`, `credp delete [NAME] [--keep-volumes]`, `credp apply [NAME]`, `credp inspect [NAME]`, `credp config [NAME] [--declared]`, `credp logs [NAME]`, `credp use NAME`, `credp create NAME`, `credp list [FILTER]`, `credp binding {add|remove|list|test} ...` — all resolve to the canonical workspace command with the current default as the implicit workspace. An explicit NAME overrides the default.

**Harness commands** — `credproxy dev ...`, for hacking on credproxy itself; need the repo checkout:

- `credproxy dev build` — `docker build` the proxy image.
- `credproxy dev test [-- PYTEST_ARGS...]` — pytest inside the proxy image. Trailing args pass through to pytest.
- `credproxy dev reload [NAME]` — SIGHUP the workspace's proxy; python re-execs in place, picking up edited source from the bind-mounted `proxy/`. The container, netns, iptables rules, and tmpfs all survive, so pushed config persists. A python crash takes the container down (no supervisor); recover with `credproxy workspace NAME start`.

**Safety gate** (loose surface only): destructive commands that operate on an **implicit** (defaulted) workspace require confirmation. Explicit targets never prompt. `--yes` / `-y` bypasses. Fails closed without a TTY. Destructive set: `delete`, `binding remove` (both high recovery cost), and `recreate --reset-volume` (wipes a managed volume's data). `stop` is not gated (restart is cheap), and neither is a plain `recreate` (all managed volumes, config, token, and state survive — only the rebuildable container is lost); the gate is flag-conditional for `recreate`, firing only with `--reset-volume`.

## Open design questions

Surface these rather than picking silently if your work touches one:

- **Injector authoring contract (resolved, fully implemented).** Injection is a typed, scheme-aware request transform: an injector selects a `scheme` (+ `params`) the proxy implements, replacing the original `format`/substring-replace. The `format` field is gone. All of it now ships: substitute + sign (`sigv4`, plus `ovh`/`jwt-bearer` as scripts) + re-seal (`oauth2-reseal` + `mint` primitives, TTL'd runtime layer), and the Starlark escape hatch on the flat/implicit-ctx "option B" primitive API (`api`-versioned manifests). See `docs/injectors.md`; don't reintroduce `format` or a single-mechanism substitution, and don't revert the scripted API to explicit ctx-threading.
- **Per-request vs. per-host injection.** Currently strictly per-host; no path/method matching. Host *scoping* does support glob patterns (`*.amazonaws.com`) as of the hostmatch work, but matching is still hostname-only — no path, method, or query matching, and a pattern can't be enumerated to a fixed SNI set (so the intercept decision is the `creds.intercepts(sni)` predicate, not membership on `intercept_hosts()`).
- **Proxy CA persistence across recreate.** mitmproxy's CA is regenerated on a proxy *recreate* (not on stop/start or a crash — those keep the same container). Currently a re-bootstrap cost, not silent breakage: a proxy recreate is coupled to a workspace recreate, and the workspace's CA trust lives only in its writable layer, so the workspace re-bootstraps anyway. Would become a real gap only if CA trust were made durable across workspace recreates.
- **`/llms.txt` format.** Currently free-form prose; structured/AGENTS.md-style alternatives haven't been evaluated.
- **Mounts are a typed model.** `mounts` entries are host **binds** (`"SRC:DST[:ro]"` strings), managed **volumes** (`{volume=NAME, target}` tables — namespaced per workspace, image-seeded, ownership-clean), or **profile** mounts (`{profile=REL, target}` — a path under the profile overlay dir, confined, read-only by default, for shipping static files with a fork). `home = "/path"` is **sugar** for a managed volume named `home`; omit it for an ephemeral home (the deferred "optional persistent home", now delivered). The chown of runtime-fabricated mount-parents is host-bind-aware (safe under any managed volume, never under a host bind), not anchored on home. Volume lifecycle: created on first `start`, survive `recreate`/`stop`, removed on `delete` (unless `--keep-volumes`), individually wiped by `recreate --reset-volume NAME`.
- **Distribution constants not customizable (by choice, for now).** The proxy image tag (`IMAGE_TAG`) is the one hardcoded engine constant; the default *workspace* image lives only in `workspace.template.toml` (and `home` is optional sugar — no fallback). A profile overlay can customize the scaffold + definitions but not `IMAGE_TAG`. If per-distribution proxy-image rebranding is wanted later, the clean path is a `proxy_image` workspace field (`ImageEnv.load(image)` already takes one) rather than reviving a `profile.toml`.
- **Build a workspace image from a Dockerfile (idea, parked — revisit when a fat-`setup` pain or an org asks).** Let a workspace declare a `build` (Dockerfile + context, mutually exclusive with `image`); `start` runs `docker build` and feeds the resulting image ID into the existing recreate path (build-on-start, recreate on image-ID change — docker's layer cache makes repeat builds cheap). **Does not break "BYO image, never modified"**: credproxy builds the user's instructions verbatim, injects nothing, runs it unprivileged — a declarative image, not a new trust model. The **netns boundary partitions the work cleanly**: the build runs *before* the container joins the proxy netns, so build egress is un-proxied and `proxy.local`/the CA aren't reachable — therefore **Dockerfile = static/public/un-credentialed provisioning; `setup` = run-time/credentialed (the proxy injects creds) + the CA bootstrap (only reachable at run-time)**. So **refuse build-time secrets** (`--build-arg`/BuildKit) on purpose — credentialed installs belong in `setup`. Composes with the profile overlay (ship a `Dockerfile`, reuse the `profile:` context confinement) and with `recreate --reset-volume home` (the home volume seeds from the image only on first create, so Dockerfile changes to the home dir need a home reset — a known sharp edge). Costs: a managed built image to lifecycle (tag `credproxy-ws-<name>`, prune on `delete`), build latency + a new failure surface, rebuild-trigger fuzziness (start with always-build, optimize later). Verdict: a convenience+lifecycle layer (`docker build -t … && image = …` already works by hand), low architectural risk, moderate surface (build orchestration + image lifecycle + error reporting).
- **Composability knobs — attach credproxy to externally-managed containers (design proposal, not implemented — see `docs/composability.md`).** Lets credproxy ride alongside another container manager (devcontainers/Compose, CI, a hand-run proxy) instead of owning the workspace. Core idea: a workspace *name* bundles a `(bindings source, proxy target, state location)` triple; the knobs make each addressable independently — `push` (extract the resolve+POST step into a first-class, foreign-targetable command: `--proxy`/`--admin`, `--wait`/`--timeout`, owned lock, atomic/fail-closed; `start`/`apply` then call this engine), `resolve` (resolve to a wire-config blob without pushing — `--json`/`--out` — for deliver-at-creation), `init`/`state-dir` (provision/print host state, i.e. the token, decoupled from container creation; `--dir`/`--print-env`), plus optional `emit-compose` scaffolding and a future `daemon` (watch `docker events` → auto `push --wait`). Proxy gains a **two-signal split**: `/health` (capture-ready, creds-agnostic — what `start`/`push --wait` poll) vs new `/ready` (creds-ready — what an external health-gate like Compose `depends_on: service_healthy` consumes). Load-bearing invariants: `--wait` polls `/health` not `/ready` (else `push→ready→creds→push` deadlock); never collapse `/health` into requiring creds (breaks standalone `start` the same way); push is atomic/fail-closed; secrets stay transient by default (`resolve --out` is the at-rest escape hatch — 0600, under state dir, never in the repo); parent-agnostic boundary (credproxy owns mechanics, the integration owns orchestration policy incl. backgrounding; discovery selectors are generic, `--compose-project`/`emit-compose` the only Compose-aware sugar); port resolved at call time; CA trust pulled by the consumer via the open bootstrap route, not pushed into a foreign trust store. Open: command placement (top-level verbs vs a `proxy`/`attach` noun group — leaning top-level) and whether to ship `resolve --out` at all given the at-rest concern.
